## Supply-chain note — laravel-lang mass-compromise (May 22–23, 2026)
An attacker republished ~700 malicious historical version tags across four `laravel-lang/*` localization packages; an injected `helpers.php` autoloads on every request. **None of those packages appear in your SBOM** — so you are not directly exposed via this path. Still worth confirming your `composer.lock` integrity hashes and that you pin/verify dependencies, since the attack rewrote historical tags (version number alone is not trustworthy for affected packages).

## Addendum (2026-07-01) — three components missed in the original pass

The first version of this triage covered `laravel/framework` and the Symfony components most visibly tied to HTTP handling. It **did not check** `league/commonmark`, `guzzlehttp/psr7`, or `symfony/yaml` — a gap that only surfaced because Composer's own security-advisory blocking (on by default since Composer 2.9) refused to install these exact SBOM-pinned versions while building the fuzzing harness. That's a good illustration of why `./scan.sh` — which checks all 91 components against a real advisory database — is the authoritative source, not either triage pass by hand.

### league/commonmark 2.6.0 — 3 advisories, all extension-gated
All three below only matter if the app has enabled the specific optional CommonMark extension named — the default converter configuration (no extensions beyond core) isn't affected by any of them.

| CVE | Issue | Fixed in | Gate |
|---|---|---|---|
| CVE-2025-46734 | XSS via crafted attribute injection | 2.7.0 | Requires `AttributesExtension` enabled |
| CVE-2026-30838 | `DisallowedRawHtml` sanitizer bypassed by whitespace in tag names (e.g. `<script\n>`) | 2.8.1 | Requires `DisallowedRawHtmlExtension` used to sanitize untrusted markdown |
| CVE-2026-33347 | `Embed` extension domain-allowlist bypass (`youtube.com.evil` matches `youtube.com`) → SSRF/XSS | 2.8.1 | Requires `EmbedExtension` enabled |

**Action:** confirm which CommonMark extensions the app actually registers. If none of the three above are in use, this is low urgency; if any are, upgrading to 2.8.1+ closes all three at once.

### guzzlehttp/psr7 2.7.0 — 3 advisories, all in raw-message parse/serialize
None of these affect the common case of using PSR-7 only through Guzzle's own HTTP client (`guzzlehttp/guzzle`, also in this SBOM at 7.9.2) — they require the app to manually parse or (re)serialize raw HTTP messages: custom proxies, webhook relays, crawlers, non-Guzzle transports.

| CVE | Issue | Fixed in |
|---|---|---|
| CVE-2026-48998 | Host confusion via authority reinterpretation in `Message::parseRequest()` / `ServerRequest::fromGlobals()` — can misdirect routing/allowlist/credential decisions | 2.10.2 |
| CVE-2026-49214 | CRLF injection via URI host component on serialize | 2.10.2 |
| CVE-2026-55766 | CRLF injection in HTTP start-line (method/version/reason-phrase) on serialize — disclosed 2026-06-23, about a week old as of this writing | 2.12.1 |

**Action:** grep the codebase for `Message::parseRequest`, `Message::parseResponse`, `Message::toString`, `ServerRequest::fromGlobals`, or `getUriFromGlobals`. If none of those are called directly (i.e. everything goes through Guzzle's client as normal), this is low urgency. If any are, upgrade to 2.12.1+.

### symfony/yaml 7.2.0 — 3 DoS advisories, all from one 2026-05-20 disclosure batch
All three require `Yaml::parse()` / `Yaml::parseFile()` to run on attacker-influenced input. **None of these are fixed by a same-line patch bump** — the fix landed only in 7.4.12+ and 8.0.12+, a different (newer) minor/LTS line than 7.2.x.

| CVE | Issue |
|---|---|
| CVE-2026-45133 | Stack exhaustion via unbounded recursion in nested blocks/sequences/mappings |
| CVE-2026-45304 | "Billion laughs" — exponential memory growth via recursive collection-alias expansion |
| CVE-2026-45305 | ReDoS via catastrophic backtracking in the `%YAML` directive-header regex |

**Action:** confirm whether the app ever parses YAML that isn't fully trusted (config files you control aren't a concern; anything from an upload, an API payload, or a partner integration is). If so, this needs the jump to 7.4.x, not just a patch release — worth scoping as its own upgrade rather than bundling with the other Symfony patches above. The fuzzing harness's `symfony_yaml` target now has seeds built directly from these three mechanics and may reproduce them independently.
